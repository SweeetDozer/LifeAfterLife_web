# Frontend Audit

## Scope

Goal: prepare the frontend for migration to the current backend API without a large refactor, without redesign, and without changing the backend contract.

## Current Project Structure

Root:

- `package.json` - Vite/React/TypeScript app, build via `tsc -b && vite build`
- `tsconfig.json`, `tsconfig.node.json` - strict TypeScript config
- `vite.config.ts` - minimal Vite config, no aliases or proxy
- `dist/`, `vite.config.js`, `vite.config.d.ts`, `*.tsbuildinfo` - build/compiler artifacts, not part of app architecture

`src/`:

- `app/` - app shell
  - `providers/AppProviders.tsx` - `QueryClientProvider` + `BrowserRouter` + `SessionProvider`
  - `routes/AppRoutes.tsx` - route table
  - `layouts/AppLayout.tsx` - main shell, nav, session status
- `pages/` - page containers
- `features/auth/` - auth API, store, guards, form UI
- `entities/` - domain modules for tree/person/relationship/kinship
- `shared/api/` - API client, React Query config, errors, generated schema
- `shared/types/api.ts` - aliases over generated OpenAPI types
- `shared/ui/` - reusable UI primitives

## How Key Parts Work Now

### Routing

- Uses `react-router-dom` v6 with `BrowserRouter`
- Routes live in `src/app/routes/AppRoutes.tsx`
- Protected pages are wrapped with `RequireAuth`
- `/login` and `/register` are wrapped with `RedirectIfAuthenticated`
- Main routes:
  - `/dashboard`
  - `/trees/:treeId`
  - `/persons/:personId`
  - `/relationships`
  - `/kinship`

Assessment: routing is simple and reusable, but still page-driven and thin.

### Auth

Current flow:

- login/register requests are in `src/features/auth/api/auth-api.ts`
- access token is stored in Zustand with `persist`
- route guard checks only `accessToken` presence
- API client injects `Authorization: Bearer <token>`
- on `401`, token is cleared and `SessionProvider` redirects to `/login`

Why this is outdated:

- access token is stored in persistent client storage
- no refresh-token flow
- no `me/session/profile` request to validate the session after hydration
- `RequireAuth` trusts token presence only
- auth state is modeled as UI strings instead of explicit session states

Assessment: acceptable as a temporary integration stub, but auth is one of the main rewrite candidates before full backend alignment.

### API Client / Server Requests

The transport layer is already separated reasonably well:

- `src/shared/api/client.ts` - shared `openapi-fetch` client
- `src/shared/api/result.ts` - `unwrap` helper
- `src/shared/api/errors.ts` - error normalization
- `src/shared/api/query-client.ts` - React Query defaults
- `src/shared/api/query-keys.ts` - cache keys

Requests are split by module:

- `features/auth/api/auth-api.ts`
- `entities/tree/api/tree-api.ts`
- `entities/person/api/person-api.ts`
- `entities/relationship/api/relationship-api.ts`
- `entities/kinship/api/kinship-api.ts`

Assessment: this part is reusable and is a solid base for reconnecting the UI to the current backend API.

### Generated OpenAPI Schema / Types

Current files:

- `src/shared/api/generated/schema.ts` - generated by `openapi-typescript`
- `src/shared/types/api.ts` - local aliases over generated `components` and `paths`

Current generation script:

```json
"generate:api": "openapi-typescript http://localhost:8000/openapi.json -o src/shared/api/generated/schema.ts"
```

How generation works now:

1. Pulls `openapi.json` from `http://localhost:8000/openapi.json`
2. Generates TypeScript schema/types only
3. Runtime client is not generated; `openapi-fetch` is typed from generated `paths`
4. App code mainly imports aliases from `src/shared/types/api.ts`

Important notes:

- generation is hardcoded to local backend URL
- there is no separate documented workflow for schema refresh
- frontend code is tightly coupled to the currently generated endpoints and DTOs
- generated schema includes `authorization` and `token` headers for some endpoints, but client code only sets `Authorization`

## Modules Better Rewritten

- `src/features/auth/model/auth-store.ts`
- `src/features/auth/model/session-context.tsx`
- `src/features/auth/ui/RequireAuth.tsx`
- `src/features/auth/ui/RedirectIfAuthenticated.tsx`
- page-level flows in `src/pages/*`, especially:
  - `DashboardPage`
  - `TreePage`
  - `RelationshipsPage`
  - `KinshipPage`

Why: these areas still behave like a technical CRUD playground over the old API rather than a stable app flow over a current backend contract.

## Where The App Is Hard-Coupled To The Old API

Literal endpoint coupling exists in:

- `/auth/register`
- `/auth/login`
- `/trees/`
- `/persons/`
- `/persons/tree/{tree_id}`
- `/persons/{person_id}`
- `/relationships/`
- `/graph/path`
- `/kinship/`

Main coupling points:

- `src/features/auth/api/auth-api.ts`
- `src/entities/tree/api/tree-api.ts`
- `src/entities/person/api/person-api.ts`
- `src/entities/relationship/api/relationship-api.ts`
- `src/entities/kinship/api/kinship-api.ts`
- `src/pages/*`, where UI maps directly to current request/response shapes

Extra signs of old-contract coupling:

- `TreePage` builds `PersonCreate` directly in backend DTO format
- `RelationshipsPage` and `KinshipPage` act like manual ID tools driven by query string `treeId`
- `DashboardPage` and `TreePage` directly depend on fields like `tree_id`, `person_id`, `access_level`, `photo_url`, `middle_name`
- there is almost no adapter/view-model layer between backend DTOs and UI rendering

## Where Auth Uses An Outdated Pattern

- `src/features/auth/model/auth-store.ts` - persisted `accessToken` storage
- `src/features/auth/ui/RequireAuth.tsx` - auth check by token presence only
- `src/shared/api/client.ts` - bearer token injection from client store
- `src/features/auth/model/session-context.tsx` - session handling is mostly UI redirect on `401`

This is not necessarily wrong for a temporary local frontend, but it is not a robust long-term auth model for an actively evolving backend API.

## Dead Code / Secondary Noise

- No clearly dead application module was found from the current structure
- Root contains generated/build artifacts: `vite.config.js`, `vite.config.d.ts`, `tsconfig*.tsbuildinfo`, `dist/`
- These are noise during audit, but removing them is outside the scope of this task

## Summary

The project is already in better shape than a typical legacy frontend:

- centralized API client exists
- OpenAPI types already exist
- structure follows `app / features / entities / shared`
- React Query is already in place

The main problems are not in the base infrastructure. The main problems are:

- auth is too simplistic
- pages are too directly tied to current DTOs and endpoints
- there is no adaptation layer between backend contract and UI

## Next 3 Logical Steps

1. Regenerate and verify the schema from the actual current backend API, without changing frontend code by guesswork.
2. Add a thin adapter layer between generated DTOs and UI models so pages stop depending on raw backend response shapes.
3. Rebuild auth around the real backend flow: token/session storage, current-user validation, `401` handling, and route guarding.

## Commands For Run / Check

```bash
npm run dev
npm run build
npm run generate:api
```

Note: `npm run generate:api` currently requires a backend available at `http://localhost:8000/openapi.json`.
